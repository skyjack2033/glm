# Revised Plan: One Physical Wireless Unified Output Assembly

## 必须满足的新约束

用户明确要求：**一个物理方块同时被 GT 当成 Output Bus 和 Output Hatch**。

因此计划不再拆成两个公开方块。最终公开给玩家的只有一个 MetaTileEntity / 一个放置方块：

- `Wireless Unified Output Assembly (ME)`
- 中文名：`无线统一输出总成 (ME)`
- 这个方块在 GT 多方块结构里同时提供：
  - 物品输出总线能力，即 GT 侧能把它当 `MTEHatchOutputBus`
  - 流体输出仓能力，即 GT 侧能把它当 `MTEHatchOutput`

## 当前调研结论

你的实际 GTNH 2.9.0-beta-1 实例版本：

- GregTech: `gregtech-5.09.52.594.jar`
- AE2: `appliedenergistics2-rv3-beta-977-GTNH.jar`
- ModularUI2: `modularui2-2.3.80-1.7.10.jar`
- GTNHLib: `gtnhlib-0.11.24.jar`
- UniMixins: `+unimixins-all-1.7.10-0.3.1.jar`

当前项目依赖要先对齐：

- `GT5-Unofficial:5.09.52.594:dev`
- `Applied-Energistics-2-Unofficial:rv3-beta-977-GTNH:dev`

当前项目主要问题：

- 现在的 `MTEWirelessOutputHatchME` 继承 `MTEHatchOutput`，所以 GT 原生只把它识别为流体输出仓。
- 当前 `WirelessOutputBusAdapter` 是普通 Java 对象，不是 `MTEHatchOutputBus`，不适合作为 `mOutputBusses` 成员。
- 当前 first tick 扫描控制器并反射写入 `mOutputBusses` 是脆弱的：结构可能还没形成，GT `clearHatches()` 后会丢失，且 `getOutputBusses()` 会过滤有效 MTE。
- 当前 `long` 容量字段未真实实现，只有 hardcoded `Long.MAX_VALUE`。
- 当前绑定逻辑仍混着 Memory Card / WAP / controller coordinates 命名，和“自定义绑定工具”目标不一致。

AE2 调研结论：

- `AEApi.instance().createGridConnection(IGridNode, IGridNode)` 可用。
- `IGridNode.setPlayerID(int)` 可用。
- 玩家权限应保存 `UUID/name`，加载时用 `WorldData.instance().playerData().getPlayerID(GameProfile)` 重算 player ID。
- 绑定工具可以检测：
  - `appeng.tile.networking.TileController`
  - `appeng.tile.misc.TileSecurity`
- 第一版用本模组自定义绑定工具，不再修改 AE2 Memory Card 行为。

## 核心架构：单物理方块，双 GT 视图

GT 的输出侧没有输入侧那种原生 `IDualInputHatch`。要让一个物理方块同时进入 `mOutputBusses` 和 `mOutputHatches`，采用这个结构：

```text
World block / BaseMetaTileEntity
    |
    |-- real MTE: MTEWirelessUnifiedOutputAssemblyME extends MTEHatchOutputBus
    |       - 被 GT 原生识别为 Output Bus
    |       - 负责玩家交互、绑定工具、GUI、NBT、AE2 proxy、item output
    |
    |-- delegate MTE: WirelessFluidOutputDelegate extends MTEHatchOutput
            - 不是独立方块，不注册物品形式
            - 共享 real MTE 的 base tile
            - 被注入/附加进同一个控制器的 mOutputHatches
            - 负责把 fluid output 委托给 real MTE 的 output core
```

这个方案的关键点：

- 物品侧完全走 GT 原生 `MTEHatchOutputBus` 路径。
- 流体侧用一个真正的 `MTEHatchOutput` delegate，而不是普通 adapter。
- 玩家只看到、放置、绑定一个方块。
- 不恢复 GT class Mixin。
- 反射只用于把 delegate 放入当前控制器的 `mOutputHatches`，且对象类型是真正的 `MTEHatchOutput`，比当前普通 adapter 可靠。
- 需要周期性重新附加 delegate，因为 GT 结构重检会 `clearHatches()`。

## 风险和 fallback

主要风险：GTNH 2.9.0 是否接受一个共享 base tile 的 `MTEHatchOutput` delegate 长期留在 `mOutputHatches`。

如果 delegate 在运行时被 GT 判定无效或引发结构异常，fallback 不是恢复 Mixin，而是：

- 保留一个物理方块的外观和玩家交互。
- 运行时在结构检查后由真实 MTE 主动接管流体输出 dispatch，或
- 退到两个公开方块方案作为兼容版本。

但第一实现路线仍然以“一物理方块 + real bus MTE + fluid delegate MTE”为主。
